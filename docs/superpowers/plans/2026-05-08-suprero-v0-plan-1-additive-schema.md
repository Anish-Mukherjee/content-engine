# Suprero v0 — Plan 1: Migration Phase 1 (Additive Multi-Tenant Schema)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add tenancy + auth tables and nullable `site_id` columns to all existing pipeline tables. Deploy safely to production. content-pipeline behaves identically afterward; xerogravity.com keeps working.

**Architecture:** Additive Drizzle migration only. No FKs to existing tables, no NOT NULL on existing rows, no behavior changes in pipeline code. New tables for Better Auth core (user/session/account/verification) + organization plugin (organization/member/invitation) + Suprero-specific (site/api_call_log). Existing tables get a nullable `site_id uuid` column with no constraint.

**Tech Stack:** TypeScript, Drizzle ORM 0.45, Drizzle Kit 0.31, Postgres, Vitest, npm.

**Spec:** [`docs/superpowers/specs/2026-05-08-suprero-v0-design.md`](../specs/2026-05-08-suprero-v0-design.md)

**Why this plan defers workspace setup:** The spec mandates a shared schema package. We're deferring that to Plan 2 (when `suprero-app` first needs to import the schema). Doing it now would force a prod deploy structure change with no immediate benefit. This plan keeps content-pipeline's deploy exactly as it is.

**Why this plan defers pipeline code changes:** All pipeline code edits (threading `siteId` through stages, scheduler iteration) live in **Plan 4**, where they ship together with the Phase 4 migration that makes `site_id` NOT NULL. Until then, code reads/writes existing columns only.

### Task 2: Add a script to snapshot prod DB safely

**Files:**
- Create: `content-pipeline/scripts/snapshot-prod-db.sh`

This script is used both for the dry-run later in this plan AND for every future schema migration. Reusable, executable, gitignored output.

- [ ] **Step 1: Create the snapshot script**

```bash
cat > scripts/snapshot-prod-db.sh <<'EOF'
#!/usr/bin/env bash
# Snapshots the production content-pipeline DB to a local SQL file.
# Usage: ./scripts/snapshot-prod-db.sh [output-file]
# Default output: ./prod-snapshot-YYYYmmdd-HHMMSS.sql (gitignored)

set -euo pipefail

OUT="${1:-prod-snapshot-$(date +%Y%m%d-%H%M%S).sql}"
SSH_HOST="${PROD_SSH_HOST:-backend-vps}"
DB_USER="${PROD_DB_USER:-pipeline}"
DB_NAME="${PROD_DB_NAME:-content_pipeline}"

echo "Snapshotting $DB_NAME from $SSH_HOST → $OUT"
ssh "$SSH_HOST" "sudo -u postgres pg_dump --format=plain --no-owner --no-privileges $DB_NAME" > "$OUT"

bytes=$(wc -c < "$OUT")
echo "Snapshot complete: $OUT ($bytes bytes)"
if [ "$bytes" -lt 1000 ]; then
  echo "ERROR: snapshot suspiciously small (<1KB). Aborting."
  exit 1
fi
EOF
chmod +x scripts/snapshot-prod-db.sh
```

import { describe, it, expect, beforeAll, afterAll } from 'vitest';

describe('schema 0003 — additive multi-tenant tables', () => {

### Task 12: Rollback procedure (only run if Task 11 fails)

**Files:**
- Production DB schema is reverted

This task is **not run as part of a successful Plan 1 execution**. It's documented here so you can execute it without thinking if Task 11 fails.

- [ ] **Step 1: Drop new tables and the site_id columns**

SSH to prod, then:

```bash
ssh backend-vps
sudo -u postgres psql content_pipeline <<'SQL'
BEGIN;

-- Drop nullable site_id from existing tables (safe — column was added empty)
ALTER TABLE articles DROP COLUMN IF EXISTS site_id;
ALTER TABLE seed_keywords DROP COLUMN IF EXISTS site_id;
ALTER TABLE keyword_results DROP COLUMN IF EXISTS site_id;
ALTER TABLE dataforseo_tasks DROP COLUMN IF EXISTS site_id;
ALTER TABLE image_usage DROP COLUMN IF EXISTS site_id;

-- Drop new tables in dependency order
DROP TABLE IF EXISTS api_call_log CASCADE;
DROP TABLE IF EXISTS site CASCADE;
DROP TABLE IF EXISTS invitation CASCADE;
DROP TABLE IF EXISTS member CASCADE;
DROP TABLE IF EXISTS organization CASCADE;
DROP TABLE IF EXISTS verification CASCADE;
DROP TABLE IF EXISTS account CASCADE;
DROP TABLE IF EXISTS session CASCADE;
DROP TABLE IF EXISTS "user" CASCADE;

-- Remove the migration ledger entry
DELETE FROM drizzle.__drizzle_migrations
WHERE hash IN (
  SELECT hash FROM drizzle.__drizzle_migrations ORDER BY created_at DESC LIMIT 1
);

COMMIT;
SQL
```

Expected: `COMMIT` printed, no errors.
